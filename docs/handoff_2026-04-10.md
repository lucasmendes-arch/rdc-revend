# Handoff — Portal do Parceiro + Catálogo Avançado

_Data: 2026-04-10_
_Autor: Claude (assistido por Lucas)_
_Contexto: sessão completa de desenvolvimento — portal do parceiro, ordenação de catálogo, admin de clientes_

---

## 1. Objetivo

Implementar o portal do parceiro de rede (`network_partner`) com login por telefone, exibição de preços da tabela de preço personalizada, e controle manual de ordenação dos produtos no catálogo via drag-and-drop no admin.

---

## 2. Diagnóstico (estado antes)

- Login aceitava apenas e-mail. Parceiros da rede têm `auth_phone` mas não e-mail memorizado.
- `AuthContext.isPartner` checava só `profiles.is_partner` (boolean legado), ignorando parceiros criados com `customer_segment = 'network_partner'` e `is_partner = false`.
- `useCatalogProducts` não buscava `get_my_price_list_items()` — parceiros viam preço padrão do catálogo mesmo com tabela vinculada.
- Catálogo re-ordenava produtos alfabeticamente no frontend (`sortBy = 'name_asc'`), tornando qualquer ordenação do banco irrelevante.
- `catalog_products` não tinha `sort_order` — nenhum controle de ordem era possível.
- Popup de perfil incompleto e PackageCards apareciam para parceiros.
- Preços de parceiros exibiam o preço original riscado + preço parceiro em âmbar (layout "promoção"), quando o correto era exibir apenas um preço.
- Admin não permitia editar dados cadastrais do cliente nem alterar tabela de preço vinculada.

## 4. O que foi implementado

### Backend

**Migration 20260410000001 — `admin_update_profile`**
- RPC SECURITY DEFINER, admin-only
- Atualiza: `full_name`, `phone`, `document_type`, `document`, `business_type`, `employees`, `revenue`
- Usa COALESCE — campos NULL preservam valor existente

**Migration 20260410000002 — `resolve_partner_login_email`**
- RPC acessível por `anon` e `authenticated`
- Recebe telefone em E.164, retorna e-mail do parceiro ativo
- Condição: `auth_phone = p_phone AND customer_segment = 'network_partner' AND access_status = 'active'`
- Retorna NULL se não encontrado (sem expor erro de "usuário não existe")

**Migration 20260410000003 — `sort_order` em `catalog_products`**
- `sort_order integer NOT NULL DEFAULT 0`
- Índice em `(category_id, sort_order)`
- Seed inicial: `ROW_NUMBER() OVER (PARTITION BY category_id ORDER BY updated_at DESC) - 1`

**Migration 20260410000004 — `admin_update_product_sort_orders`**
- RPC SECURITY DEFINER, admin-only
- Recebe `jsonb` `[{"id": "uuid", "sort_order": 0}, ...]`
- Loop FOR sobre o array, UPDATE individual por produto
- Necessária porque escrita direta em `catalog_products` é admin-only via RLS e updates diretos do cliente falhavam silenciosamente

### Frontend — Auth / Hook

**`AuthContext.tsx`**
- `fetchAccountMetadata` agora seleciona `role, is_partner, customer_segment`
- `isPartner = !!data.is_partner || data.customer_segment === 'network_partner'`
- Cobre parceiros legados (`is_partner = true`) e novos (`customer_segment = 'network_partner'`)

**`useCatalogProducts.ts`**
- Nova opção: `fetchPriceList?: boolean`
- Quando `true`: executa `supabase.rpc('get_my_price_list_items')` em paralelo via segundo `useQuery`
- Merge em `useMemo`: produtos com entry na price list têm `partner_price` sobrescrito pelo preço da lista
- NaN guard: entradas com `price <= 0` ou `NaN` são ignoradas no merge
- `priceListError` exposto para que o componente possa alertar o usuário em caso de falha

**`useAdminProducts.ts`**
- Campo `sort_order?: number` adicionado ao tipo `CatalogProduct`
- Query inclui `sort_order`, ordenação por `sort_order ASC, updated_at DESC`
- Hook `useBulkUpdateSortOrder`: chama `admin_update_product_sort_orders(updates)` via RPC, invalida `['admin-products']` e `['catalog-products']` on success

### Frontend — Login por Telefone

**`Login.tsx`**
- Função `looksLikePhone(value)`: retorna `true` se 10–13 dígitos após strip de `[\s\-\(\)\+\.]`
- Função `normalizePhone(raw)`: normaliza para E.164 (`+55` prefix para 10–11 dígitos)
- Fluxo de telefone: `normalizePhone` → `rpc('resolve_partner_login_email', { p_phone })` → se retornar e-mail → `signInWithPassword({ email, password })` → navigate `/catalogo`
- "Esqueci minha senha" oculto quando `identifierIsPhone = true`
- Campo visualmente idêntico ao login de e-mail (label "E-mail", placeholder `seu@email.com`) — comportamento muda silenciosamente

### Frontend — Catálogo

**`Catalogo.tsx`**
- `useCatalogProducts({ includePartnerPrice: isPartner, fetchPriceList: isPartner })`
- Popup de perfil incompleto: guard adicional `&& !isPartner` no useEffect e na dependency array `[user?.id, role, isPartner]`
- PackageCards: envoltos em `{!isPartner && (...)}` (mobile e desktop)
- `sortBy` default alterado de `'name_asc'` para `'default'` — preserva ordem do banco
- Novo case `'default'` no switch: retorna `result` sem re-ordenar
- Aviso em âmbar exibido quando `priceListError && !error` (apenas para parceiros com lista falhando)
- Opção "Padrão" adicionada aos selects de ordenação (mobile e desktop)
- Badge de filtro ativo: `sortBy !== 'default'` (era `!== 'name_asc'`)

**`CompactProductCarousel.tsx`**
- Badge "Uso Profissional" (violet/roxo, ícone `Briefcase`) quando `product.is_professional = true`
- Badges "Vegano" + "Liberado" apenas quando `is_professional = false`
- Preço parceiro: exibe só `R$ {partner_price}` sem preço original riscado

### Frontend — Admin

**`admin/Clientes.tsx`**
- `selectedSessionId: string | null` em vez de `selectedSession: ClientSession | null`
- `selectedSession` derivado live de `sessions.find(s => s.id === selectedSessionId)` — elimina bug de estado stale após mutação
- Seção de tabela de preço: `AdminSelect` com listas ativas, visível apenas para `customer_segment = 'network_partner'`
- Edição inline de perfil: estado `editingProfile`, formulário com todos os campos do cadastro, botão Editar/Salvar/Cancelar
- `updateProfileMutation` via RPC `admin_update_profile`
- Imports: `Edit2`, `Check` adicionados

**`admin/Catalogo.tsx`**
- Botão "Reordenar" no header → dropdown com categorias (state-controlled, não hover CSS)
- Clique em categoria → `openReorder(categoryId)`: filtra produtos ativos da categoria, ordena por `sort_order`, abre modal
- Modal: `DndContext` + `SortableContext` + `SortableProductRow` (componente separado acima do `export default`)
- `handleDragEnd`: `arrayMove` no state local
- `handleSaveOrder`: `bulkUpdateSortOrder.mutateAsync([{id, sort_order: index}])` → `reorderSaved = true`
- Imports dnd-kit: `DndContext`, `closestCenter`, `PointerSensor`, `KeyboardSensor`, `useSensors`, `SortableContext`, `useSortable`, `verticalListSortingStrategy`, `arrayMove`, `CSS`

## 5. Contrato de dados

### Login por telefone
```ts
// Frontend detecta telefone
looksLikePhone(identifier) // true se 10-13 dígitos

// RPC resolve e-mail
supabase.rpc('resolve_partner_login_email', { p_phone: '+5511999999999' })
// → string (e-mail) | null

// Login normal com e-mail retornado
supabase.auth.signInWithPassword({ email: partnerEmail, password })
```

### Price list no catálogo
```ts
// Hook com merge automático
const { data: products, priceListError } = useCatalogProducts({
  includePartnerPrice: isPartner,
  fetchPriceList: isPartner,
})
// products[n].partner_price já está sobrescrito pelo preço da lista para o parceiro logado
// priceListError: Error | null — exibir aviso se não-null
```

### Sort order
```ts
// Salvar nova ordem (admin)
supabase.rpc('admin_update_product_sort_orders', {
  updates: [{ id: 'uuid', sort_order: 0 }, { id: 'uuid', sort_order: 1 }]
})

// Query do catálogo (automático via useCatalogProducts)
.order('sort_order', { ascending: true })
.order('updated_at', { ascending: false })
```

### Edição de perfil (admin)
```ts
supabase.rpc('admin_update_profile', {
  p_user_id: 'uuid',
  p_full_name: 'Nome',
  p_phone: '11999999999',
  p_document_type: 'cpf',
  p_document: '000.000.000-00',
  p_business_type: 'MEI',
  p_employees: '1-5',
  p_revenue: '5k-20k',
})
```

### `auth_phone` para login por telefone
O campo `profiles.auth_phone` deve estar preenchido em E.164 (`+55XXXXXXXXXXX`) e `access_status = 'active'` para que o login por telefone funcione. Gerenciado pela edge function `admin-partner-credentials`.

## 6. Pendências

- `partner_price` não tem proteção column-level para usuários `authenticated` não-parceiros. Apenas a lógica frontend condiciona o select. Seria necessário um GRANT revoke ou view separada para garantia completa.
- Badge "Uso Profissional" adicionado apenas no `CompactProductCarousel` e no grid flat do `Catalogo.tsx`. Verificar se há outros lugares de exibição de produto (ex: detalhe de produto, carrinhos) que precisem do mesmo badge.
- O campo `sort_order` é global por produto, não por `(category_id, sort_order)` — se um produto mudar de categoria, o sort_order absoluto pode gerar ordem inesperada. A UI de reordenação já trabalha por categoria, minimizando o risco, mas não há constraint de unicidade por categoria.

## 7. Riscos / Observações

**Login por telefone — dependência de `auth_phone`**
Se `auth_phone` não estiver preenchido ou não estiver em E.164, o login por telefone retorna "E-mail ou senha incorretos" (RPC retorna NULL). O admin deve garantir que o campo seja preenchido corretamente ao criar credenciais do parceiro.

**isPartner no auth — loading race**
Existe um window curto durante o carregamento onde `isPartner = false` antes de `fetchAccountMetadata` completar. Neste período, `useCatalogProducts` é chamado com `fetchPriceList: false`. Quando `isPartner` vira `true`, a query key muda (`['catalog-products', true]`) e uma nova query é disparada automaticamente. O efeito visual é um brief de preços padrão antes dos preços da lista aparecerem — aceitável.

**sort_order e sincronização com Nuvemshop**
A migration `sync-nuvemshop` faz upsert em `catalog_products`. O `sort_order` não é sobrescrito pelo sync (campo não existe na Nuvemshop), então a ordenação manual sobrevive a sincronizações. Confirmar que a edge function de sync não tem `UPDATE ... SET sort_order = ...` em nenhum path.

**Drag-and-drop — touch em iOS/Android**
`@dnd-kit` com `PointerSensor` funciona em touch, mas pode ter conflito com scroll. Se reportado, adicionar `TouchSensor` com `activationConstraint: { delay: 250, tolerance: 5 }`.

**Feature Freeze — `create-order`**
Nenhuma alteração nos arquivos em freeze. Não há impacto no fluxo de checkout.
